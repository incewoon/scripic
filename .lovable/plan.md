문제는 사용자의 "마무리해줘"가 감지된 뒤에도, 실제 앨범 생성 함수가 최신 대화 상태를 못 받아 실행이 건너뛰는 흐름일 가능성이 큽니다. 아래처럼 수정하겠습니다.

1. 채팅 종료 트리거 흐름 재점검
- `src/routes/chat.tsx`에서 명시 종료 문구 감지 → AI 응답 스트리밍 완료 → `finish()` 호출까지의 순서를 다시 확인합니다.
- 특히 `setMessages(...)` 직후 `finish()`가 예전 `messages` state를 참조하는지 점검합니다.

2. 최신 메시지 기준으로 앨범 생성 보장
- `finish()`가 React state의 지연 반영에 의존하지 않도록, 방금 완성된 대화 배열을 직접 넘길 수 있게 정리합니다.
- 사용자가 종료를 요청한 턴과 방금 받은 AI 응답까지 포함된 최신 transcript로 `aiGenerateAlbum()`이 실행되게 만듭니다.

3. 종료 조건을 더 안전하게 분리
- "AI가 마무리 제안함", "사용자가 긍정함", "사용자가 직접 마무리 요청함" 세 경우를 분리해서 처리합니다.
- 단순 동의 응답만 하고 멈추는 케이스를 막기 위해, 명시 종료 요청은 AI 토큰 유무와 무관하게 최종 생성 경로로 강제 연결합니다.

4. 중복 실행/미실행 방지
- `finishingRef`와 `generating` 가드를 정리해서
  - 두 번 생성되는 문제는 막고
  - 한 번도 생성이 안 되는 문제도 없게 조정합니다.

5. 서버 프롬프트 일관성 확인
- `functions/src/prompts-chat.ts`의 `[READY_TO_FINISH]` 규칙은 유지하되,
- 클라이언트가 토큰 없이도 명시 종료 요청을 처리하도록 역할을 분명히 나눕니다.

6. 검증
- 스크린샷과 같은 케이스("테스트야 마무리해줘")를 기준으로
  - 명시 종료 요청
  - 짧은 긍정 응답(네/ㅇㅋ/그래)
  - 영어 finish/wrap up 류
  를 각각 확인합니다.

기술 메모
- 핵심 수정 파일은 우선 `src/routes/chat.tsx`입니다.
- 필요 시 `functions/src/prompts-chat.ts`는 최소 범위로만 조정합니다.
- 목표는 UI 문구 변경이 아니라, 종료 감지 후 실제 앨범 생성 호출이 빠지지 않도록 흐름을 안정화하는 것입니다.
