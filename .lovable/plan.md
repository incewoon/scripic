## 확인한 현재 동작

`src/routes/chat.tsx`의 마무리 트리거는 다음 한 가지 경로로만 발동합니다.

1. **직전** 어시스턴트 메시지에 `[READY_TO_FINISH]` 토큰 또는 `WRAP_HINT_*` 정규식이 포함되어 있어야 `wrapProposed = true`
2. 그 위에 사용자의 답변이 `AFFIRMATIVE_*` 정규식과 매치되어야 `userAgreed = true`
3. 둘 다 충족 → 새 어시스턴트 스트림 끝난 뒤 600ms 후 `finish()` 호출

서버 프롬프트(`functions/src/prompts-chat.ts`)는 "사진×턴 상한에 도달했을 때"만 `[READY_TO_FINISH]`를 붙이도록 지시합니다.

## 발견한 문제

A. **사용자가 먼저 "마무리해줘 / 정리해줘 / wrap it up" 등 명시적 요청을 할 때 무시됨**
- AI가 아직 `[READY_TO_FINISH]`를 낸 적이 없으면 `wrapProposed=false` → 트리거 자체가 안 걸림
- 그래서 AI는 "알겠어요, 정리할게요" 같은 응답만 하고 실제 `finish()`로 안 넘어감 (사용자가 보고한 정확한 증상)

B. **방금 스트리밍이 끝난 어시스턴트 응답 안의 `[READY_TO_FINISH]` 토큰은 체크하지 않음**
- 트리거 판정은 스트림 시작 전 `prior`의 마지막 어시스턴트만 봄
- 사용자 요청에 반응해 AI가 새 응답 끝줄에 토큰을 붙이는 케이스에서도 자동 finish가 안 일어남

C. **`AFFIRMATIVE_KO` 정규식이 너무 넓음**: `해주세요|해줘|부탁|괜찮` 포함 → 일반 대화에서도 매치되어 직전 턴에 `[READY_TO_FINISH]`가 와 있던 상황에서는 의도치 않은 finish가 발생할 수 있음

D. **`WRAP_HINT_KO`도 다소 넓음**: 부분 일치(`마무리해`)라 사진 설명 문맥에서도 우발 매치 가능

E. **서버 프롬프트**: 사용자가 턴 상한 전에 명시적으로 마무리/정리/완성을 요청하면 즉시 `[READY_TO_FINISH]`를 붙이라는 지시가 없음 → 위 A의 백업 경로도 없는 상태

## 변경 계획

### 1) 사용자 명시적 마무리 요청 패턴 추가 (`src/routes/chat.tsx`)
"확실한 마무리 의도" 전용 정규식을 새로 만들어, 직전 wrap proposal 여부와 무관하게 `finish()`를 예약합니다.

- 한국어: `(앨범|이걸|이거)?\s*(마무리|정리|완성|마감|끝내)\s*(해|해줘|해주세요|해주실|할래|할까|하자|부탁)`,  단독 "끝내줘 / 마무리 / 정리해 / 완성해"
- 영어: `\b(finish|wrap up|wrap it up|finalize|complete|put it together|create the album)\b.*\b(now|please)?\b` 류

매치되면 `prior`의 wrap 제안 여부와 관계없이 `userAgreed = true` 로 취급.

### 2) "방금 스트리밍된" 어시스턴트 응답도 트리거 대상에 포함
스트림이 끝난 뒤 새 어시스턴트 텍스트(`assistant`)에 `[READY_TO_FINISH]` 또는 wrap hint가 포함되어 있으면, 사용자 추가 입력 없이도 `finish()`를 예약. (단, 사용자가 1번 케이스로 이미 명시 요청한 경우 중복 호출 방지)

### 3) `AFFIRMATIVE_KO` / `WRAP_HINT_KO` 좁히기
- AFFIRMATIVE_KO에서 `해주세요|해줘|부탁|괜찮` 제거. 유지: `네|예|넵|응|어|그래(요)?|좋아(요)?|ㅇㅇ|ㅇㅋ|오케이|콜|끝|완성|마무리|충분|됐어|그래그래`
- WRAP_HINT_KO는 어시스턴트의 "정리/마무리 제안" 문장 패턴으로 좁힘: `(앨범으로 (정리|마무리)|이대로 (정리|마무리)|정리할까요|마무리할까요|완성할까요|정리해 ?드릴까요|마무리해 ?드릴까요|완성해 ?드릴까요)` — 단독 부분일치 제거
- 또한 wrap proposal 판정에서 `[READY_TO_FINISH]` 토큰이 가장 신뢰도 높은 신호이므로 토큰만 있어도 통과시키는 현재 로직 유지

### 4) 서버 프롬프트 보강 (`functions/src/prompts-chat.ts`)
`turnLimitClause`에 다음 조항 추가 (한/영 모두):
- "사용자가 명시적으로 앨범 마무리/정리/완성/끝내달라고 요청하면, 턴 상한 도달 여부와 관계없이 즉시 짧게 동의 응답한 뒤 마지막 줄에 정확히 `[READY_TO_FINISH]` 토큰을 붙여라."
- 이로써 클라이언트 1)과 2) 모두에 안전망 제공

### 5) 트리거 안정화 / 사이드이펙트 점검
- `finishingRef`는 finally에서 풀리지만, 트리거가 두 경로(사용자 명시 / 응답 내 토큰)에서 발생할 수 있으므로 둘 다 동일한 `if (!finishingRef.current)` 가드 안에서 처리
- `messages.length < 2` 가드는 그대로 두되, 명시적 마무리 트리거에서 메시지 너무 적으면 `t.talkMore` 토스트만 띄우고 트리거 해제

## 검증 시나리오

- 한국어 정상 흐름: 턴 상한 도달 → AI가 토큰 포함 제안 → "네" / "ㅇㅇ" → 자동 finish (기존 회귀 없음)
- 한국어 조기 종료: 턴 상한 전에 사용자가 "마무리해줘" → AI 짧은 동의 + 토큰 → 자동 finish (새 동작)
- 한국어 조기 종료 (서버 미지원 모델 응답): 사용자 "정리해줘" → AI 토큰 없이 "알겠어요" → 그래도 클라이언트 1)이 직접 트리거 → finish (새 동작)
- 영어 동등 케이스 (`finish it`, `wrap it up`)
- 일반 대화에서 "사진 설명 부탁해요", "그 부분 괜찮아요" 같은 문장이 잘못 트리거 안 되는지 (3 변경의 결과)
- READY 토큰 화면 표시 제거 유지 확인 (`replaceAll(READY_TOKEN, "")`)
